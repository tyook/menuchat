# Capacitor Mobile App — Design Spec

**Date:** 2026-03-30
**Status:** Approved

## Overview

Wrap the existing Next.js frontend in Capacitor to produce native Android and iOS apps from the same codebase. Both customers and restaurant owners use a single app. Includes push notifications, native QR code scanning, and over-the-air live updates via Capawesome.

## Section 1: Project Structure & Build Pipeline

### New Files & Directories

```
frontend/
  src/                    # (unchanged)
  android/                # Generated by `npx cap add android`
  ios/                    # Generated by `npx cap add ios`
  capacitor.config.ts     # Capacitor configuration
  next.config.mjs         # Add output: 'export'
  package.json            # Add capacitor deps + build scripts
```

### Build Flow

1. `next build` produces static files in `frontend/out/`
2. `npx cap sync` copies `out/` into the native projects (`android/` and `ios/`)
3. Open in Android Studio / Xcode to run on device or build for store

### New npm Scripts

| Script | Command |
|--------|---------|
| `build:mobile` | `next build && npx cap sync` |
| `open:android` | `npx cap open android` |
| `open:ios` | `npx cap open ios` |
| `deploy:mobile` | Builds, zips, and uploads bundle for live updates |

### API URL Handling

- `NEXT_PUBLIC_API_URL` already defaults to `http://localhost:5005`
- For mobile builds, set to production backend URL at build time
- During development, point at local machine IP (e.g., `http://192.168.x.x:5005`)
- Same applies to `NEXT_PUBLIC_WS_URL` for WebSocket connections

### Next.js Config Change

```js
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export',
};
export default nextConfig;
```

Note: `output: 'export'` is only needed for mobile builds. The web deployment can continue using the default Next.js build. Consider using an environment variable to toggle this.

## Section 2: Native Plugin Integration

### Push Notifications (`@capacitor/push-notifications`)

- On app launch, request notification permission and register with APNs (iOS) / FCM (Android)
- Send the device token to Django backend via `POST /api/accounts/devices/`
- Backend stores device tokens and sends notifications via Firebase Cloud Messaging (works for both platforms)
- **Notification triggers:**
  - Customer: order status changes (confirmed, ready, etc.)
  - Kitchen/Owner: new incoming orders
- Web users do not receive push notifications (can add web push later)

### QR Code Scanner (`@capawesome/capacitor-mlkit-barcode-scanning`)

- New "Scan QR" button in app navigation (visible on native only)
- Opens native camera, reads QR code URL (e.g., `https://yourapp.com/order/slug/table3`)
- Parses the URL and navigates to the correct `/order/[slug]/[tableId]` route internally
- Falls back gracefully on web (button hidden)

### Platform Detection

New utility at `src/lib/platform.ts`:

```ts
import { Capacitor } from '@capacitor/core';

export const isNative = () => Capacitor.isNativePlatform();
export const getPlatform = () => Capacitor.getPlatform(); // 'ios' | 'android' | 'web'
```

Used to conditionally show/hide native-only features (QR scan button, push notification registration).

## Section 3: CORS & Auth Changes

### CORS — Backend (`config/settings.py`)

Add Capacitor origins to the allowed origins list:

- `capacitor://localhost` (iOS)
- `http://localhost` (Android)

These are required for both development and production — Capacitor always makes requests from these origins regardless of where the backend is hosted.

### Authentication

- Current cookie-based JWT auth with CSRF tokens works in Capacitor's WebView
- Keep this approach for now
- If auth issues arise on specific devices/WebView versions, add `Authorization: Bearer` header fallback — the backend already issues JWT tokens, so this would be a minimal change (read from localStorage instead of cookies)

### WebSocket URL

- Same as API URL — needs to point at deployed backend for production
- During dev, use machine's local IP

## Section 4: What Stays the Same

- All React components, hooks, stores, types — unchanged
- Tailwind CSS / shadcn/ui styling — works as-is in Capacitor WebView
- Google OAuth, Stripe — work in WebView (may need OAuth redirect URI config for Capacitor scheme)
- Next.js routing — static export handles dynamic routes client-side
- The web version continues to be deployed and used independently

## Section 5: Live Updates (Capawesome)

### How It Works

- On app launch, the Capawesome live update plugin checks a server for a newer bundle
- If a new version exists, it downloads in the background and applies on next app restart
- Users always get the latest web code without visiting the App Store

### Infrastructure

- Host bundle zips on any static file server (S3, Cloudflare R2, or Django serving static files)
- A simple manifest JSON file tracks the current version
- No paid service needed

### Update Flow

1. Make code changes and run `next build`
2. Zip the `out/` directory, upload to bundle server with a version ID
3. Update the manifest to point at the new version
4. Users' apps pick it up automatically on next launch

### What Still Requires a Store Submission

- Adding or removing Capacitor native plugins
- Changing `capacitor.config.ts`
- Updating native project files (`android/`, `ios/`)
